## 1. Общее описание проекта

Необходимо разработать кроссплатформенное приложение на Flutter для личного планирования, организации дня и контроля задач пользователя с хаотичным графиком.

Основной сценарий использования — журналист, у которого:

- много рабочих и личных событий;
- есть дедлайны, встречи, интервью, поездки, звонки;
- есть проблемы с тайм-менеджментом и планированием;
- есть потребность быстро понимать, когда он свободен;
- есть риск забывать о важных делах и мероприятиях.

Приложение должно объединять задачи, события и свободные интервалы в одном удобном интерфейсе и помогать пользователю видеть свой день целиком.

## 2. Цель MVP

Первая версия должна решить базовые проблемы пользователя:

- хранение и управление задачами;
- хранение и управление событиями;
- отображение плана дня в одном месте;
- показ мест, где пользователь должен быть сегодня;
- отображение просроченных задач;
- показ свободных окон для встреч;
- напоминания о важных задачах и событиях;
- работа без интернета;
- корректная работа как на мобильных устройствах, так и на компьютерах.

## 4. Основные требования к реализации

### 4.1. Без авторизации

На первом этапе приложение работает без регистрации и без входа в аккаунт.

Все данные пользователя хранятся локально на устройстве.

### 4.2. Offline-first

Приложение должно полноценно работать без интернета.

### 4.3. Масштабируемая архитектура

На первом этапе необходимо заложить качественную, чистую, поддерживаемую архитектуру, на основе которой в будущем можно будет безболезненно добавить:

- авторизацию;
- облачную синхронизацию;
- интеграцию с календарями;
- AI-функции;
- шаринг свободных слотов;
- аналитику.

### 4.4. Архитектурный подход

Обязательное требование: использовать GetX для:

- state management;
- dependency injection;
- навигации.

При этом проект должен быть организован по принципам Clean Architecture или максимально приближенной к ней модульной архитектуры.

## 5. Концепция приложения

Приложение — это персональный планировщик/ассистент, который помогает пользователю:

- видеть все дела и события на день;
- не забывать о важных задачах;
- контролировать дедлайны;
- понимать, где именно ему нужно быть в течение дня;
- понимать свободные интервалы;
- быстро переносить невыполненные задачи;
- удобно работать как с телефона, так и с компьютера.

## 5.1. Карта мест

Картография входит в продукт поэтапно и не считается отдельной, изолированной фичей вне MVP.

Назначение карты:

- показать, где пользователю нужно быть сегодня;
- отображать события на карте;
- позже отображать и задачи, если у них есть место выполнения.

Поэтапный ввод:

- сначала карта питается событиями;
- затем поддержка мест добавляется и для задач;
- в интерфейсе появятся и отдельный экран карты, и компактный блок на экране “Сегодня”.

Минимальный картографический контракт:

- у события сохраняется текстовое место;
- для отображения маркера используются optional `latitude` и `longitude`;
- позже задачи получают такой же optional location-contract.

Если у объекта есть только текстовое место без координат, он остается в списках и карточках, но не получает маркер на карте.

## 7. Функциональные требования MVP

### 7.1. Главный экран “Сегодня”

Экран должен показывать:

- текущую дату;
- краткую сводку дня;
- ближайшее событие;
- блок “Где мне быть сегодня”;
- список событий на сегодня;
- список задач на сегодня;
- блок просроченных задач;
- блок свободных окон;
- кнопку быстрого добавления.

**Поведение:**

- события отображаются по времени;
- задачи с фиксированным временем идут в хронологическом порядке;
- задачи без времени выводятся отдельным списком;
- просроченные задачи выводятся отдельным блоком;
- события с координатами могут выводиться в компактном картографическом блоке;
- события без координат остаются в текстовом списке дня;
- выполненные задачи могут отображаться внизу списка или скрываться по переключателю.

### 7.2. Создание задачи

Пользователь должен иметь возможность создать задачу.

**Обязательные поля:**

- название;
- дата.

**Необязательные поля:**

- описание;
- время;
- длительность;
- приоритет;
- категория;
- флаг “на весь день”.

**Действия:**

- сохранить задачу;
- отменить создание.

### 7.3. Редактирование задачи

Пользователь должен иметь возможность:

- изменить любые поля задачи;
- отметить задачу выполненной;
- перенести задачу на другую дату;
- удалить задачу.

### 7.4. Создание события

Пользователь должен иметь возможность создать событие.

**Обязательные поля:**

- название;
- дата;
- время начала;
- время окончания.

**Необязательные поля:**

- описание;
- место;
- координаты места (`latitude`, `longitude`);
- категория;
- контактное лицо;
- контакт;
- напоминание.

### 7.5. Редактирование события

Пользователь должен иметь возможность:

- изменить событие;
- удалить событие.

### 7.6. Экран календаря

Должен быть отдельный экран календаря.

**Функции:**

- отображение календаря по месяцам;
- выбор даты;
- отображение индикаторов дней, на которых есть задачи/события;
- показ списка элементов выбранного дня.

### 7.7. Экран “Все задачи”

Экран должен содержать:

- список всех задач;
- фильтрацию;
- сортировку;
- быстрые действия.

**Фильтры:**

- режим `На дату / Все`;
- по статусу.

**Сортировка:**

- по времени;
- по приоритету;
- по дате создания.

В текущем Sprint 1 именно этот экран уже реализован в базовом виде: есть режим `На дату / Все`, выбор даты для режима `На дату`, фильтр по статусу и простая сортировка.

Расширенные фильтры по категории, приоритету и дедлайну остаются следующим слоем.

### 7.8. Экран “Свободное время”

Экран предназначен для быстрого ответа на вопрос: когда пользователь свободен.

**Функции:**

- выбор даты;
- отображение свободных интервалов;
- показ длительности каждого интервала;
- исключение слишком коротких слотов на основе настроек.

**Логика:**

- рабочий диапазон дня берется из настроек;
- все события считаются занятыми интервалами;
- задачи с фиксированным временем также считаются занятыми;
- свободные интервалы вычисляются автоматически.

### 7.9. Напоминания

Приложение должно поддерживать локальные уведомления.

Напоминания работают для:

- задач;
- событий.

Пользователь должен иметь возможность:

- включить/выключить напоминание;
- выбрать время напоминания.

**Базовые варианты:**

- в момент начала;
- за 10 минут;
- за 30 минут;
- за 1 час;
- за 1 день.

### 7.10. Просроченные задачи

Эта логика входит в следующий продуктовый слой и пока не реализована в коде.

На текущем этапе задача имеет только статусы `pending` и `completed`, а отдельный `overdue`-сценарий запланирован вместе с календарным спринтом.

### 7.11. Быстрое добавление

На главном экране должна быть кнопка быстрого добавления.

После нажатия пользователь выбирает:

- создать задачу;
- создать событие.

Для MVP используется стандартная форма.
Обработка естественного языка не требуется.

### 7.12. Настройки

Необходимо реализовать экран настроек.

**Минимальные настройки:**

- начало рабочего дня;
- конец рабочего дня;
- минимальная длина свободного окна;
- включение/выключение уведомлений;
- светлая/темная/системная тема.

Дополнительно уже реализованы:

- секция “О приложении”;
- runtime version/build metadata;
- безопасный `reset settings` с подтверждением.

### 7.13. Карта мест

В продукте должен быть отдельный экран карты мест.

Первый источник данных для карты:

- события на сегодня, у которых есть координаты.

Второй источник данных:

- задачи с координатами, когда поддержка location-contract будет добавлена и для `Task`.

**Функции карты:**

- показать, где пользователю нужно быть сегодня;
- отобразить маркеры событий с координатами;
- сохранить текстовое место у сущностей, даже если координат нет;
- не терять объекты без координат: они остаются в списках и блоках, но не попадают на карту.

## 8. Адаптивность интерфейса

Так как приложение должно работать и на телефонах, и на компьютерах, интерфейс должен быть адаптивным.

### 8.1. Мобильная версия

Для мобильных устройств использовать:

- нижнюю навигацию;
- компактные карточки;
- формы в виде отдельных экранов или bottom sheet.

### 8.2. Десктопная версия

Для десктопных устройств использовать:

- боковую навигацию или rail navigation;
- более широкие экраны;
- двухколоночный или трехколоночный layout там, где это оправдано;
- отображение большего количества информации на одном экране.

**Пример:**

На десктопе экран “Сегодня” может состоять из:

- левой колонки с календарем или навигацией;
- центральной колонки с событиями и задачами;
- правой колонки со свободными окнами и ближайшими напоминаниями.

## 9. Пользовательские сценарии

### Сценарий 1

Пользователь утром открывает приложение и видит:

- план на сегодня;
- ближайшие события;
- задачи;
- свободные интервалы;
- просроченные задачи.

### Сценарий 2

Пользователь добавляет событие “Интервью” на 14:00–15:00.
Событие появляется в календаре и в списке на день.

### Сценарий 3

Пользователь добавляет задачу “Подготовить вопросы”.
Задача появляется в списке задач на выбранную дату.

### Сценарий 4

Пользователь открывает экран “Свободное время” и сразу видит, когда с ним можно встретиться.

### Сценарий 5

Пользователь не выполнил задачу вчера.
Сегодня задача отображается как просроченная, и приложение предлагает перенести ее.

## 10. Бизнес-логика

### 10.1. Правила сортировки на экране “Сегодня”

**Порядок отображения:**

- ближайшие события;
- задачи с конкретным временем;
- задачи без времени;
- просроченные задачи отдельным блоком;
- выполненные задачи в конце или по скрываемому блоку.

### 10.2. Правило просроченности

Задача считается просроченной, если:

- текущая дата больше даты задачи;
- статус не `completed`.

### 10.3. Расчет свободных окон

**Алгоритм:**

- взять рабочий диапазон дня;
- собрать все занятые интервалы;
- отсортировать интервалы по времени начала;
- объединить пересекающиеся интервалы;
- вычислить промежутки между ними;
- исключить интервалы короче минимального лимита;
- вернуть итоговый список слотов.

### 10.4. Валидация событий

Событие нельзя сохранить, если:

- время окончания раньше времени начала;
- отсутствуют обязательные поля.

### 10.5. Валидация задач

Задачу нельзя сохранить, если:

- не заполнено название;
- не выбрана дата.

### 13.4. Роль GetX в архитектуре

GetX использовать для:

- Bindings — регистрация зависимостей;
- Controllers — управление UI-состоянием;
- GetMaterialApp — маршрутизация;
- Get.find() — доступ к зависимостям через DI.

**Важно:**

GetX не должен превращать проект в “все в контроллерах”.

Бизнес-логика должна быть вынесена в:

- repository interfaces;
- services/domain layer;
- отдельные use case классы там, где это действительно оправдано.

Контроллеры должны:

- получать данные;
- вызывать repository/services/domain actions;
- управлять состоянием страницы;
- не содержать тяжелую бизнес-логику.

### 13.5. Хранение данных

Так как приложение работает без авторизации и без облака, необходимо локальное хранение.

Выбранное foundation-решение:

- Isar

На текущем этапе `Isar` уже используется для foundation-настроек и задач. Хранение событий будет добавляться поверх той же storage-инфраструктуры.

Необходимо хранить:

- задачи;
- события;
- настройки.

Для событий дополнительно резервируются optional координаты `latitude` и `longitude`, чтобы позже подключить карту без переделки модели.

### 13.6. Уведомления

На текущем foundation-этапе в приложении уже есть custom in-app overlay-уведомления для пользовательских действий и ошибок.

Для локальных уведомлений использовать:

- `flutter_local_notifications`

Нужно обеспечить:

- создание уведомления при сохранении задачи/события;
- обновление уведомления при редактировании;
- удаление уведомления при удалении сущности;
- корректную работу на мобильных платформах.

На десктопе поведение уведомлений зависит от платформенных возможностей; базовая поддержка приветствуется, но критичным приоритетом MVP являются мобильные уведомления.

Настройка темы должна сохраняться локально.

Разработчик должен:

- соблюдать единый стиль кода;
- использовать понятные имена классов и методов;
- избегать дублирования логики;
- выносить повторяющиеся элементы в переиспользуемые виджеты;
- разделять UI, бизнес-логику и доступ к данным;
- не смешивать хранение, вычисления и отображение в одном месте;
- писать код так, чтобы новые функции можно было добавлять без переписывания половины проекта.

## 18. Критерии приемки

MVP считается выполненным, если:

- Приложение запускается на Android, Windows и Linux.
- Пользователь может создать, редактировать и удалить задачу.
- Пользователь может создать, редактировать и удалить событие.
- Пользователь может отметить задачу как выполненную.
- Просроченные задачи отображаются отдельно.
- На экране “Сегодня” отображаются актуальные данные.
- На экране “Сегодня” есть блок “Где мне быть сегодня”.
- На экране календаря можно выбрать дату и увидеть элементы дня.
- На экране “Свободное время” корректно рассчитываются интервалы.
- Экран карты мест показывает события с координатами.
- Объекты без координат не теряются и остаются доступны в текстовых списках и карточках.
- Данные сохраняются локально после перезапуска приложения.
- На Android работают локальные уведомления.
- Интерфейс адаптирован под телефон и под компьютер.
- Архитектура проекта организована с использованием GetX и понятного разделения по слоям/модулям.
