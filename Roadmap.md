# Roadmap проекта

**Актуально на:** 17 марта 2026

**Текущее состояние roadmap:** Sprint 0 закрыт как foundation-этап. В коде уже есть архитектурный каркас, shell/navigation, theme/palette system, settings storage, logging/errors, custom in-app notifications, desktop polish и locale/date-time foundation. Бизнес-модули MVP еще не начаты.

## Общая последовательность

## Платформы по версиям

### Первая версия

- Android
- Windows
- Linux

### Вторая версия

- Web

### Третья версия

- iOS
- macOS

### Этап 1. Фундамент

Сначала создаем архитектуру, проектный каркас и базовые правила разработки.

### Этап 2. Базовые сущности

Потом реализуем ядро: задачи, события, локальное хранение.

### Этап 3. Основной пользовательский сценарий

После этого собираем “Сегодня”, календарь и редактирование сущностей.

### Этап 4. Умная логика MVP

Затем добавляем просроченные задачи, свободные окна, настройки.

### Этап 5. Платформенная готовность

Потом — адаптация под desktop/mobile, уведомления, UX-доработки.

### Этап 6. Стабилизация и релиз

И только после этого — багфикс, тестирование, подготовка сборок.

## Sprint 0 — Подготовка и архитектурный фундамент

**Статус:** реализован

### Цель

Подготовить проект так, чтобы дальше разработка шла быстро, чисто и без хаоса.

### Что было запланировано

- финализируем scope MVP;
- фиксируем список экранов;
- утверждаем основные сущности: `Task`, `Event`, `Settings`, `FreeSlot`;
- проектируем архитектуру на GetX + Clean/feature-based structure;

**Создаем Flutter-проект под:**

- Android
- Windows
- Linux

**Настраиваем:**

- базовую структуру папок;
- базовые зависимости;
- routing через GetX;
- dependency injection через Bindings;
- theme system;
- storage abstraction;
- базовую обработку ошибок;
- линтеры и стиль кода;
- адаптивный каркас UI для mobile/desktop;
- пустые страницы основных модулей.

### Что фактически уже собрано

- feature-first каркас `app / core / features`;
- bootstrap приложения, app-scoped DI и routing на `GetX`;
- `MainLayout` с `SidebarX`, `GNav`, `PageView` и placeholder-экранами основных разделов;
- theme system: `system / light / dark` + palette;
- foundation-настройки с локальным хранением на `Isar`;
- app-level logging, базовая обработка ошибок и custom in-app overlay-уведомления;
- `intl`, фиксированная локаль `ru_RU` и единый formatter для даты и времени;
- desktop polish: shortcuts/intents, focus traversal, safe narrow desktop behavior;
- базовые unit/widget tests для foundation-слоя.

### Результат спринта

- проект запускается на Android, Windows и Linux;
- навигация между основными экранами уже существует;
- архитектурный каркас собран и готов к бизнес-модулям;
- есть база для масштабирования.

### Картография на Sprint 0

Карта в Sprint 0 еще не реализуется в коде, но на этом этапе фиксируется архитектурное направление:

- будущий map-модуль будет отдельной feature;
- будущий route: `/map`;
- foundation для картографии: `flutter_map`;
- первым источником данных для карты станут события;
- позже к карте подключаются и задачи.

### Почему это первый спринт

Если этого не сделать сразу, дальше проект начнет “расползаться”, и потом переделка будет стоить дорого.

## Sprint 1 — Модуль задач (Task Core)

### Цель

Реализовать первую полноценную бизнес-сущность — задачи.

### Что делаем

- создаем domain/entity/model для `Task`;
- создаем repository interface и local repository implementation;
- подключаем локальное хранилище;

**Реализуем use cases:**

- `create task`
- `update task`
- `delete task`
- `get tasks by date`
- `get all tasks`
- `mark task completed`
- `reschedule task`

**Создаем:**

- экран списка задач;
- экран создания/редактирования задачи;
- валидацию формы;
- фильтрацию и сортировку в простом виде;

**Добавляем состояния:**

- `empty`
- `loading`
- `error`

Покрываем критичную логику базовыми тестами.

### Результат спринта

Пользователь уже может:

- создавать задачи;
- редактировать задачи;
- удалять задачи;
- отмечать выполненными;
- видеть список задач.

### Это важнее всего после Sprint 0

Потому что задачи — самое базовое ядро приложения. Без них нельзя проверить ценность продукта.

## Sprint 2 — Модуль событий (Event Core)

### Цель

Добавить вторую ключевую сущность — события.

### Что делаем

- создаем entity/model для `Event`;
- создаем repository и local data source;

**Реализуем use cases:**

- `create event`
- `update event`
- `delete event`
- `get events by date`

**Создаем экран создания/редактирования события.**

**Реализуем форму события:**

- название
- дата
- время начала
- время окончания
- место
- optional `latitude` / `longitude`
- категория
- контакт
- напоминание

**Дополнительно:**

- добавляем валидацию времени;
- делаем список событий на выбранную дату;
- прорабатываем хранение и обновление событий.
- закладываем картографический контракт: текстовое место сохраняется всегда, координаты добавляются опционально.

### Результат спринта

Пользователь уже может:

- создавать события;
- редактировать их;
- удалять;
- хранить рабочие и личные встречи отдельно от задач.

### Зависимость

Этот спринт должен идти после задач, потому что структура будет похожая, и команда закрепит архитектурный подход.

## Sprint 3 — Экран “Сегодня” и объединение сценариев

### Цель

Собрать вместе задачи и события в главный ежедневный сценарий.

### Что делаем

- реализуем экран “Сегодня”;

**Добавляем:**

- текущую дату;
- ближайшее событие;
- блок “Где мне быть сегодня”;
- события на сегодня;
- задачи на сегодня;
- задачи с временем;
- задачи без времени;

**Реализуем:**

- правила сортировки;
- блок просроченных задач;

**Добавляем быстрые действия:**

- отметить задачу выполненной;
- перенести задачу;
- открыть на редактирование;

**Также:**

- делаем FAB / quick add;
- делаем базовый dashboard controller/use case composition;
- оптимизируем отображение для mobile и desktop.
- готовим компактный блок мест: события с координатами могут быть показаны как картографическая сводка дня.

### Результат спринта

После этого появляется уже первая реально полезная версия продукта:
пользователь открывает приложение и видит свой день.

### Контрольная точка

После Sprint 3 уже можно показать Alpha v0.1.

## Sprint 4 — Календарь и логика просроченных задач

### Цель

Добавить удобную навигацию по дням и завершить основной сценарий планирования.

### Что делаем

- создаем экран календаря;
- реализуем месячный вид;
- показываем индикаторы дней с задачами и событиями;
- по нажатию на дату показываем список элементов дня;

**Дорабатываем логику просроченных задач:**

- автоматическое определение `overdue`;
- отдельный блок;

**Добавляем быстрые действия:**

- перенести на сегодня
- перенести на завтра
- выбрать дату

**Также:**

- улучшаем фильтры задач;
- выравниваем UX между Today, Calendar и Tasks.

### Результат спринта

Пользователь может:

- смотреть дела не только на сегодня, но и на любую дату;
- быстро разбирать хвосты;
- планировать день через календарь.

### Почему это не раньше

Потому что календарь без готовых задач и событий — просто пустая оболочка.

## Sprint 5 — Свободные окна и настройки

### Цель

Добавить главную “ассистентскую” фишку MVP — понимание, когда пользователь свободен.

### Что делаем

- создаем сущность/модель настроек;
- реализуем локальное хранение настроек;

**Создаем экран настроек:**

- начало рабочего дня
- конец рабочего дня
- минимальная длина свободного окна
- тема
- уведомления on/off

**Реализуем use case расчета свободных окон.**

**Учитываем:**

- события;
- задачи с фиксированным временем;
- рабочий диапазон;
- минимальную длину слота;

**Создаем и добавляем:**

- экран “Свободное время”;
- отображение интервалов по выбранной дате;
- тесты на алгоритм расчета интервалов.

### Результат спринта

Появляется одна из главных ценностей приложения:
можно быстро понять, когда пользователь реально свободен.

### Контрольная точка

После Sprint 5 — собран основной planning-core MVP, если не считать уведомления, картографию и финальную полировку.

## Sprint 6 — Уведомления, адаптивность и UX-полировка

### Цель

Сделать продукт пригодным к повседневному использованию.

### Что делаем

- подключаем локальные уведомления;
- настраиваем создание уведомлений для задач и событий;
- обновляем уведомления при редактировании;
- удаляем уведомления при удалении сущностей;
- проверяем поведение уведомлений на Android;

**Улучшаем desktop UX:**

- боковая навигация / rail
- многоколоночные layout
- работа при resize окна
- корректные отступы и адаптивность

**Добавляем:**

- пустые состояния
- удобные сообщения об ошибках
- skeleton/loading
- confirm dialogs для удаления
- отдельный экран `/map`
- markers по сегодняшним событиям с координатами
- подключение задач с координатами как второго источника карты

Дорабатываем тему и визуальную консистентность.

### Результат спринта

Приложение становится уже не просто “рабочим”, а достаточно удобным для регулярного использования.

## Sprint 7 — Стабилизация, тестирование и подготовка к первой версии

### Цель

Подготовить первую релизную версию без архитектурного мусора и очевидных багов.

### Что делаем

- регрессионное тестирование;
- исправление багов;
- проверка CRUD-сценариев на всех платформах;
- проверка сохранения данных между перезапусками;
- проверка адаптивности на разных размерах окон и экранов;

**Проверяем edge cases:**

- пустые списки
- некорректные даты
- пересекающиеся события
- длинные названия
- удаление последнего элемента

**Также:**

- чистка технического долга;
- ревизия структуры проекта;
- актуализация README;
- описание архитектуры;
- описание сборки проекта;
- подготовка demo build.

### Результат спринта

Готова MVP v1.0, которую уже можно тестировать в реальной жизни.

## Приоритеты по важности

### Критически важное

Это должно быть сделано в первую очередь:

- архитектурный каркас;
- локальное хранение;
- задачи;
- события;
- экран “Сегодня”.

### Очень важное

Следующий слой ценности:

6. календарь;
7. просроченные задачи;
8. свободные окна;
9. настройки;
10. карта мест.

### Завершающее

Финальная эксплуатационная готовность:

11. уведомления;
12. desktop/mobile адаптация;
13. полировка и стабилизация.

## Что не трогаем до MVP

Чтобы не распыляться, в backlog после первой версии сразу уносим:

- авторизацию;
- облачную синхронизацию;
- интеграцию с Google Calendar;
- AI-ассистента;
- голосовой ввод;
- NLP-ввод задач;
- аналитику;
- экспорт/импорт;
- совместную работу.

Это важно, потому что такие вещи легко “съедают” сроки и ломают фокус команды.

## Definition of Done для каждого спринта

Для каждого спринта задача считается завершенной, если:

- функционал реализован;
- код встроен в общую архитектуру;
- нет хардкода бизнес-логики в UI;
- есть базовая валидация;
- feature работает на mobile и desktop;
- есть демо сценарий;
- обновлена документация по модулю;
- нет критических багов по данному функционалу.

## Самый практичный порядок по неделям

Если упростить до очень прикладного плана, то идти лучше так:

- Спринт 0: каркас проекта
- Спринт 1: задачи
- Спринт 2: события
- Спринт 3: экран “Сегодня”
- Спринт 4: календарь + просроченные
- Спринт 5: свободные окна + настройки
- Спринт 6: уведомления + карта + адаптивность
- Спринт 7: тестирование + релиз
